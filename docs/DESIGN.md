# 设计文档（DESIGN）

本文档描述本项目的架构设计与关键设计决策，帮助你快速理解：系统由哪些模块构成、为什么这么选、遇到的问题如何解决，以及后续可演进方向。

## 1. 目标与范围

### 目标
- 提供一个本地可运行的 RAG 示例：把本地文档向量化、检索、注入上下文并完成问答。
- 提供对话历史：按会话落盘，便于回放与调试。
- 提供“长期记忆”能力：从新增用户发言抽取结构化事实（name / relation / observation），去重更新并存储到向量库（可选写入本地 JSON 图谱）。
- 提供可视化验证页面：对话输入、记忆查询、记忆管理、结构化返回（回答 + 引用记忆）。

### 非目标（本阶段暂未做）
- 企业级权限体系、多租户隔离（目前仅以 `session_id` 做隔离）。
- 完整的生产级可观测性（日志、指标、追踪）。
- 真正的图数据库（Neo4j/TigerGraph）与复杂图算法（目前先用 JSON 图谱模拟结构与流程）。

## 2. 技术栈选择与理由

### Streamlit（前端）
- 理由：开发效率高，适合快速验证“对话 + 检索 + 管理”的闭环；对 Python 生态友好。
- 取舍：界面能力与性能不如完整前端框架，但用于 PoC/验证非常合适。

### LangChain（编排框架）
- 理由：内置 Prompt、Runnable、消息历史、向量库集成等能力，能快速组合出 RAG 链路与记忆抽取链路。
- 取舍：抽象层较多，调试需要一定成本，因此在实现中尽量保持“模块边界清晰、可单测”。

### Tongyi / DashScope（LLM 与 Embeddings）
- 理由：与项目现有代码一致，且同时提供对话模型与 embedding 模型。
- 取舍：需要 API Key，网络与配额会影响稳定性；因此项目内提供了不调用外部 LLM 的自测脚本用于验证核心逻辑。

### Chroma（向量数据库，本地持久化）
- 理由：本地落盘、零服务依赖、与 LangChain 集成成熟，适合示例项目与本地迭代。
- 取舍：在更复杂场景需要多机扩展、权限与 SLA 时，应替换为更成熟的向量数据库服务。

### 本地 JSON 图谱（Graph 模拟）
- 理由：项目当前未引入图数据库依赖，先用 JSON 文件实现“节点/边结构 + 去重更新 + 元数据”，便于后续替换为真正图数据库。
- 取舍：查询与一致性能力有限，仅用于验证数据结构与写入流程。

## 3. 架构设计思路

### 3.1 模块划分

项目按“职责单一 + 可替换”的原则拆成几层：

1) **UI 层**
- `app.qa.py`：RAG 问答页面（基于对话输入 + RAG 链路）。
- `app.memory.py`：记忆系统验证台（对话输入、记忆查询、记忆管理，输出结构化结果）。

2) **对话/RAG 编排层**
- `rag.py`：组装检索器、prompt、模型、输出解析，并挂载消息历史（`RunnableWithMessageHistory`）。
- `vector_stores.py`：封装 Chroma 向量库实例并提供 retriever。

3) **历史存储层**
- `file_history_store.py`：把消息历史按 `session_id` 写入本地文件；在 `add_messages` 时触发记忆抽取（副作用，失败不影响主流程）。

4) **记忆系统（抽取 + 去重 + 存储）**
- `memory_service.py`
  - `PromptMemoryExtractor`：提示词工程抽取结构化 JSON（name/relation/observation）
  - `ChromaMemoryStore`：写入记忆向量库（独立 collection），按主键 upsert
  - `JsonGraphMemoryStore`：可选写入 JSON 图谱
  - `MemoryService`：编排入口（只处理新增 HumanMessage）

### 3.2 数据流（从用户输入到回答 + 引用记忆）

以 `app.memory.py` 为例，核心链路如下：

1) 用户在 UI 输入一句话（user_text）
2) 记忆抽取：
   - `MemoryService.process_new_messages(...)` 从本轮新增的 `HumanMessage` 抽取结构化记忆
   - 按 `session_id + name + relation` 生成稳定 `memory_id`，写入 Chroma（可选写入 JSON 图谱）
3) 记忆检索：
   - 用本轮输入作为 query，在 Chroma 记忆库中按 `session_id` 过滤检索 top-k
4) 回答生成：
   - 把“引用记忆”整理成文本上下文，提供给对话模型生成回答
5) UI 输出结构化结果：
   - `{ answer: "...", citations: [ {id, text, metadata, score}, ... ] }`

### 3.3 去重与更新策略

记忆的“去重”与“更新”遵循以下原则：
- 主键：`session_id + name + relation`（规范化后生成稳定 ID）
- 更新：同一主键出现新 observation 时，尽量合并（避免丢失旧事实），并刷新 `updated_at`
- 元数据：保存 `created_at/updated_at/source/session_id/name/relation`，便于审计与回溯

## 4. 关键设计决策说明

### 决策 1：把记忆触发点放在 `file_history_store.add_messages`
- 原因：这是“新增对话消息落盘”的统一入口，最小侵入地把记忆能力接入现有 RAG 链路。
- 风险：记忆抽取是副作用，任何异常不能影响主流程。
- 解决：对记忆抽取/写入全程 try/except 包裹，失败静默或 UI 侧提示，不阻断对话与落盘。

### 决策 2：记忆向量库与 RAG 文档向量库分 collection
- 原因：避免检索混淆（文档知识与用户记忆属于不同语义空间与生命周期）。
- 做法：`memory_collection_name="memory"`（可配置），与 `collection_name="rag"` 分离。

### 决策 3：不依赖“模型一定输出纯 JSON”，而是做容错解析
- 原因：LLM 可能输出解释文字、Markdown code fence、格式不稳定。
- 做法：
  - Prompt 中强约束“只输出 JSON 数组”
  - 解析端 `_extract_json_candidate` 支持从 code fence 中提取 JSON 片段
  - `_safe_load_memories` 对异常直接返回空列表，避免崩溃

### 决策 4：用 delete + add 实现 upsert
- 原因：在示例级实现中，最稳妥地保证“同一主键只有一条记录”，并完成内容更新。
- 取舍：会产生一次删除与新增操作；当写入规模很大时可以换成更高效的批量 update/upsert 方式。

### 决策 5：Graph 先用 JSON 文件模拟
- 原因：降低依赖与安装复杂度，把“数据模型与写入逻辑”先跑通。
- 取舍：复杂查询能力不足；后续可替换为图数据库并提供更强的关系查询与推理。

## 5. 遇到的挑战与解决方案

### 挑战 1：LLM 输出格式不稳定
- 表现：输出多余文字、markdown code fence、JSON 不可解析。
- 解决：
  - 提示词严格要求“只输出 JSON 数组”
  - 解析端做容错提取与安全加载（失败返回空列表）

### 挑战 2：记忆逻辑不能影响主业务链路
- 表现：网络/模型异常可能导致对话中断或历史无法写入。
- 解决：
  - 在历史落盘后再触发记忆（先保证核心数据可用）
  - 全部记忆逻辑 try/except，异常不向外扩散
  - UI 侧捕获异常并提示

### 挑战 3：需要可运行的本地验证，不依赖外部模型
- 表现：没有 API Key 或网络时无法验证记忆去重/更新。
- 解决：
  - 提供自测脚本，使用 `FakeEmbeddings` 与假抽取器验证 upsert/合并逻辑

### 挑战 4：运行产物与缓存文件污染仓库
- 表现：`__pycache__/*.pyc`、`chroma_db/`、`chat_history/` 等不应提交。
- 解决：
  - 增加 `.gitignore` 忽略规则，避免误提交

## 6. 如果有更多时间，如何改进

### 6.1 工程化与稳定性
- 引入统一日志（结构化日志、脱敏策略）、指标与链路追踪。
- 为 `file_history_store` 增加文件锁或改用数据库/Redis，避免并发写导致覆盖。
- 把“记忆抽取/写入”改成异步队列（后台任务），减少对对话时延的影响。

### 6.2 记忆质量与策略
- 引入置信度、重要性评分、过期策略（TTL）、冲突解决策略（谁覆盖谁）。
- 从“拼接 observation”升级为结构化合并（例如：列表化事实、带时间戳、版本化）。
- 支持用户显式指令（“记住/忘记/更正”）并提供权限与审计。

### 6.3 检索与引用
- 引入更严谨的检索配置：把 `k` 与“相似度阈值”分开，支持 rerank。
- 在回答中提供更规范的引用格式（例如：按 `id` 列表引用、点击查看元数据）。

### 6.4 图谱演进
- 替换 JSON 图谱为 Neo4j 等图数据库，提供：
  - 节点/边 schema 约束
  - 基于关系的查询（例如：某人-家人-偏好）
  - 图推理或规则引擎

### 6.5 安全与密钥管理
- 统一使用环境变量或 `.env`，并确保密钥不进入 README/代码/提交历史。
- 增加启动前的配置检查（缺 key 时提示并降级到本地自测模式）。
